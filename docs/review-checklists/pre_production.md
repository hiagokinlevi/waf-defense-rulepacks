# Pre-Production WAF Deployment Checklist

Use this checklist before switching any WAF rule from log/count mode to block/enforce mode. Complete every item and document the sign-off.

---

## Section 1: Rule Review

- [ ] I have read the full pack JSON file, including `objective`, `risk_mitigated`, `potential_side_effects`, `tuning_notes`, and `deployment_notes`
- [ ] I understand what the rule blocks and why
- [ ] I have identified all application endpoints that could be affected by this rule
- [ ] I have consulted with the application team about endpoints that accept inputs similar to the rule's target patterns (e.g., search endpoints, rich text editors, data import endpoints)
- [ ] I have reviewed the `recommended_for` field and confirmed this rule is appropriate for the target application type

## Section 2: Log Mode Validation

- [ ] The rule has been deployed in log/count mode
- [ ] Log mode has been active for at least **72 hours** (or the period specified in `deployment_notes`)
- [ ] I have reviewed all events generated by the rule during the log period
- [ ] I have confirmed that all blocked events during the log period are **legitimate threat traffic** (not false positives)
- [ ] I have documented any false positive events and added appropriate exclusions

### Log Review Checklist

For each event reviewed during the log period, confirm:

- [ ] Request URI path — does this match a legitimate application endpoint?
- [ ] Request method — expected method for this endpoint?
- [ ] Source IP — known threat actor, scanner, or legitimate user?
- [ ] Request body / query string — does it contain actual attack payload or legitimate data?
- [ ] User-Agent — known scanner or legitimate browser/API client?

## Section 3: Exclusions

- [ ] All exclusions have been documented (path, reason, approver, review date)
- [ ] Exclusions are as narrow as possible (specific path + method, not broad wildcard)
- [ ] No exclusions permanently bypass the rule for entire endpoint groups without documented justification
- [ ] Exclusions have been reviewed and approved by a second person

## Section 4: Deployment Preparation

- [ ] I have notified the application team of the planned deployment date/time
- [ ] I have a rollback plan: how to quickly switch from block mode back to log mode if a false positive is discovered in production
- [ ] Monitoring dashboards are set up to detect a block rate spike after deployment
- [ ] Alert thresholds are configured (e.g., alert if block rate > 2x baseline within 5 minutes)
- [ ] On-call contact is available for the first 4 hours after deployment

## Section 5: Post-Deployment (First 24 Hours)

- [ ] Block rate after deployment was reviewed and is within expected range
- [ ] No user-reported access issues in the first 24 hours
- [ ] Security events log reviewed for any unexpected block patterns
- [ ] False positive rate is acceptably low (< 0.1% of legitimate traffic)
